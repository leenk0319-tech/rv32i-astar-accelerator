# 2026-06-23 개발일지 - RV32I A* Accelerator SoC Day 1

## 프로젝트 개요

- 프로젝트명: RV32I A* Accelerator SoC
- 목표: RV32I 기반 CPU를 직접 구현하고, 이후 MMIO bus와 A* pathfinding accelerator를 통합하는 FPGA SoC를 만든다.
- 타겟 보드: Terasic DE2-115 / Intel Cyclone IV
- 언어: Verilog HDL
- 현재 단계: CPU instruction fetch, register file, immediate generation, ALU, writeback까지 이어지는 초기 싱글사이클 CPU skeleton 검증

오늘은 전체 SoC나 5-stage pipeline을 바로 구현하지 않고, 가장 기본적인 CPU 실행 경로를 먼저 살리는 데 집중했다. 즉 `program.hex`에서 instruction을 읽고, instruction field를 잘라 decode에 필요한 값을 만들고, register file과 ALU를 거쳐 결과가 다시 register file에 writeback되는 흐름을 확인했다.

## 오늘 한 일 요약

1. RV32I subset instruction 범위를 1차로 정했다.
2. RISC-V instruction field와 opcode/funct3/funct7 decode 방식을 확인했다.
3. `imem.v`, `reg_file.v`, `imm_gen.v`, `alu.v`, `cpu_core.v`를 모듈로 나눠 작성했다.
4. `program.hex`를 만들어 instruction memory 초기값으로 사용했다.
5. `tb_cpu_core.v` testbench를 작성했다.
6. Questa 라이선스 문제 때문에 실제 실행 검증은 Vivado xsim으로 진행했다.
7. ADDI와 R-type ADD/SUB/AND/OR/XOR register writeback 테스트를 통과했다.

## RV32I Subset 결정

처음부터 RV32I 전체를 구현하지 않고, A* accelerator 제어와 기본 CPU datapath 검증에 필요한 subset으로 시작하기로 했다.

| 구분 | Instruction |
| --- | --- |
| R-type | ADD, SUB, AND, OR, XOR |
| I-type | ADDI, SLLI, XORI, ORI, ANDI |
| U-type | LUI |
| Memory | LW, SW |
| Branch | BEQ, BNE, BLT, BGE |
| Jump | JAL, JALR |

총 19개 instruction을 장기 목표로 잡았다. 다만 오늘 실제 검증 범위는 ADDI와 R-type ALU instruction으로 제한했다. 아직 LW/SW, branch, jump는 실행 경로에 완전히 붙이지 않았다.

## Instruction Decode 이해

오늘 정리한 핵심은 instruction마다 opcode를 전부 따로 두는 방식이 아니라, instruction format 단위로 opcode를 보고 funct3/funct7로 세부 연산을 구분해야 한다는 점이다.

예를 들어 ADD, SUB, AND, OR, XOR는 모두 opcode가 같다.

```verilog
OP_RTYPE = 7'b011_0011
```

따라서 ADD와 SUB는 opcode만으로 구분할 수 없고 `funct3`, `funct7`까지 봐야 한다.

```verilog
funct3 = instr[14:12]
funct7 = instr[31:25]
```

I-type ALU instruction도 마찬가지다. ADDI, SLLI, XORI, ORI, ANDI는 모두 I-type opcode를 공유하고, funct3/funct7로 세부 동작을 나눈다.

오늘 사용한 주요 field는 다음과 같다.

```verilog
wire [6:0] OP     = IR[6:0];
wire [4:0] RD     = IR[11:7];
wire [2:0] funct3 = IR[14:12];
wire [4:0] RS1    = IR[19:15];
wire [4:0] RS2    = IR[24:20];
wire [6:0] funct7 = IR[31:25];
```

## 구현 파일별 역할

### `cpu_core.v`

CPU top module 역할을 한다. 오늘 구현한 CPU 전체 데이터 흐름을 이 파일에서 연결했다.

주요 역할:

- `PC` register 관리
- `imem`에서 instruction fetch
- instruction field 분리
- `reg_file`, `imm_gen`, `alu` 인스턴스 연결
- writeback enable, writeback address, writeback data 생성

현재 PC는 단순히 매 cycle 4씩 증가한다.

```verilog
always @(posedge clk_i or negedge rstn_i) begin
    if (!rstn_i)
        PC <= 32'd0;
    else
        PC <= PC + 32'd4;
end
```

아직 branch나 jump에 따른 `pc_next` 선택 구조는 없다. 다음 단계에서 `PC + 4`, branch target, JAL target, JALR target 중 하나를 고르는 구조로 바꿔야 한다.

### `imem.v`

Instruction memory 역할을 한다.

`program.hex` 파일을 `$readmemh`로 읽어서 32-bit instruction array에 초기화한다.

```verilog
reg [31:0] mem [0:DEPTH-1];

initial begin
    $readmemh(INIT_FILE, mem);
end
```

PC는 byte address이고 RISC-V instruction은 4 byte 단위이므로 하위 2비트는 instruction index에 필요 없다. 따라서 오늘은 1024-depth instruction memory에 맞춰 `addr[11:2]`를 index로 사용했다.

```verilog
assign instr = mem[addr[11:2]];
```

처음에는 `addr[31:2]`와 `addr[11:2]` 중 무엇을 써야 하는지 헷갈렸는데, 개념적으로는 `addr[31:2]`가 byte address를 word address로 바꾸는 것이 맞다. 다만 실제 `mem` 배열이 1024개이므로 index 폭은 10비트면 충분하고, 그래서 `addr[11:2]`가 현재 구현에는 더 명확하다.

### `reg_file.v`

32개의 32-bit general-purpose register를 구현했다.

주요 설계:

- `RS1`, `RS2` 주소를 받아 조합논리로 register 값을 읽는다.
- `W_WEN`이 켜지고 `W_WA != 0`일 때 clock edge에서 write한다.
- RISC-V 규칙에 맞게 x0은 항상 0으로 유지한다.

```verilog
always @(posedge clk_i or negedge rstn_i) begin
    if (!rstn_i) begin
        for (k = 0; k < 32; k = k + 1)
            regs[k] <= 32'd0;
    end else if (W_WEN && (W_WA != 5'd0)) begin
        regs[W_WA] <= W_WD;
    end
end

assign scr1 = (RS1 == 5'd0) ? 32'd0 : regs[RS1];
assign scr2 = (RS2 == 5'd0) ? 32'd0 : regs[RS2];
```

처음에는 register file이 instruction 전체를 받아서 `rs1`, `rs2`, `rd`를 직접 잘라도 되는지 고민했지만, 최종적으로 register file은 instruction format을 모르는 단순 register bank로 두는 것이 낫다고 판단했다. instruction field 분리는 `cpu_core.v`에서 담당한다.

### `imm_gen.v`

Immediate generator 역할을 한다.

오늘은 I-type immediate와 U-type immediate만 구현했다.

```verilog
assign imm_i = {{20{IR[31]}}, IR[31:20]};
assign imm_u = {IR[31:12], 12'b0};
```

ADDI, ANDI, ORI, XORI, SLLI는 I-type immediate를 사용한다. LUI는 U-type immediate를 사용한다. 아직 SW에 필요한 S-type immediate, branch에 필요한 B-type immediate, JAL에 필요한 J-type immediate는 구현하지 않았다.

### `alu.v`

ALU는 `src1`, `src2`, `imm`, `PC`, `OP`, `funct3`, `funct7`을 입력으로 받아 `alu_out`을 만든다.

오늘 실제로 검증한 연산:

- ADDI
- ADD
- SUB
- AND
- OR
- XOR

현재 `alu.v`에는 LOAD/STORE/BRANCH/JAL/JALR case도 일부 작성되어 있지만, 오늘 CPU 전체 datapath에서는 이 instruction들이 아직 완전히 검증되지 않았다. 특히 branch와 jump는 ALU에서 target 비슷한 값을 계산하는 것만으로는 부족하고, 실제 PC update logic과 연결되어야 한다.

오늘 단계에서는 ALU가 opcode/funct3/funct7을 직접 보고 동작하도록 구성했다. 나중에 pipeline 구조로 넘어가기 전에는 decoder/control unit을 따로 분리해서 ALU에는 내부 `alu_op`만 넘기는 방식도 검토할 수 있다.

### `program.hex`

오늘 검증용 machine code 파일이다.

```text
00500093
00700113
002081b3
40118233
0041f2b3
0020e333
0020c3b3
```

의미는 다음과 같다.

```asm
addi x1, x0, 5
addi x2, x0, 7
add  x3, x1, x2
sub  x4, x3, x1
and  x5, x3, x4
or   x6, x1, x2
xor  x7, x1, x2
```

### `tb_cpu_core.v`

testbench는 clock/reset을 만들고 CPU를 일정 cycle 동안 실행한 뒤, register file 내부 값을 확인한다.

확인한 기대값:

| Register | Expected |
| --- | --- |
| x1 | 5 |
| x2 | 7 |
| x3 | 12 |
| x4 | 7 |
| x5 | 4 |
| x6 | 7 |
| x7 | 2 |

testbench에서는 다음처럼 hierarchical reference로 내부 register 값을 읽었다.

```verilog
actual = dut.U_reg_file.regs[idx];
```

이 방식은 합성용이 아니라 simulation 확인용이다.

## 헷갈렸던 점과 해결

### Quartus와 Questa/Vivado의 역할

처음에는 Quartus에서 testbench를 바로 돌리는 것으로 생각했지만, Quartus는 기본적으로 FPGA 합성/배치/타이밍 분석/보드 업로드를 위한 도구다. testbench 실행은 Questa, ModelSim, Vivado xsim 같은 simulator가 담당한다.

정리하면 다음과 같다.

| 도구 | 역할 |
| --- | --- |
| Quartus Prime Lite | Cyclone IV용 합성, fitter, timing analysis, FPGA programming |
| Questa FSE | Intel/Altera 권장 simulation tool |
| Vivado xsim | 임시 simulation 우회용으로 사용 |

Questa FSE는 설치는 되었지만 무료 license 설정 문제로 실행이 막혔다. 그래서 오늘은 Vivado xsim으로 behavior simulation을 수행했다.

### `$readmemh` 경로 문제

Vivado에서 처음 시뮬레이션했을 때 `program.hex`를 찾지 못해서 모든 register 값이 0으로 나왔다.

원인은 `$readmemh("program.hex", mem)`의 상대경로 기준이 Verilog 파일 위치가 아니라 simulator 실행 위치라는 점이었다.

해결 방법:

- `program.hex`를 xsim 실행 위치에서 접근 가능한 곳에 두거나
- `INIT_FILE`에 상대경로를 맞게 넣거나
- 임시로 절대경로를 사용한다.

이 문제를 해결한 뒤 testbench가 정상 통과했다.

### `input wire`, `output wire`, `output reg`

Verilog module port 선언에서 input은 기본적으로 wire로 볼 수 있다. output은 `assign`으로 연결하면 wire, `always` block 안에서 값을 대입하면 reg로 선언해야 한다는 점을 정리했다.

오늘 적용한 예:

- `reg_file.v`의 read output은 `assign`이므로 `output wire`
- `alu.v`의 `alu_out`은 `always @(*)` case문에서 대입하므로 `output reg`

## 현재 한계

현재 구현은 CPU 전체가 아니다. 오늘은 초기 실행 경로만 검증했다.

아직 구현 전이거나 미완성인 것:

- 5-stage pipeline register
- forwarding
- load-use stall
- branch flush
- data memory 기반 LW/SW
- branch/jump PC redirect
- MMIO bus
- A* accelerator
- interrupt/CSR
- FPGA board bring-up

특히 `alu.v`에 branch/jump 관련 case가 있더라도, 현재 `cpu_core.v`의 PC는 무조건 `PC + 4`만 수행하므로 branch/jump가 실제로 동작한다고 볼 수 없다.

## 포트폴리오 관점에서 의미

오늘 작업은 작지만 중요한 첫 단계다. CPU가 instruction을 가져오고, register file과 ALU를 연결해서 결과를 writeback하는 경로가 실제 testbench에서 검증되었다. 앞으로 pipeline, memory, MMIO, accelerator를 붙이려면 이 기본 datapath가 먼저 안정적으로 동작해야 한다.

즉 오늘 결과는 다음을 보여준다.

- RISC-V instruction field 구조를 이해했다.
- opcode/funct3/funct7 기반 decode 방향을 잡았다.
- module instantiation 방식으로 CPU 내부 블록을 연결했다.
- `$readmemh` 기반 instruction memory 초기화 흐름을 만들었다.
- testbench로 register writeback 결과를 자동 검증했다.
